# V0.5.3 — `report_publish_result` 使用服务端时间作为 publishedAt

> **状态**: 计划中
> **目标版本**: v0.5.3
> **基础版本**: v0.5.2
> **创建日期**: 2026-06-12
> **目标交付**: `report_publish_result` 成功/部分成功时忽略外部 CLI 回传的 `publishedAt`，统一改用 NextPost 服务端收到回传时的 `Date.now()` 作为 `Post.publishedAt`，解决外部 CLI 死机/断网重试导致的时间漂移问题

---

## 1. 背景与目标

### 1.1 现状

v0.2 起的 `report_publish_result` 在 `status: 'success' | 'partial'` 时，将外部 CLI 回传的 `publishedAt` 直接写入 `Post.publishedAt`：

```ts
case 'success':
  updateData.status = 'published';
  // 使用传入的 publishedAt 或当前时间
  updateData.publishedAt = publishedAt ? new Date(publishedAt) : new Date();
  ...
case 'partial':
  updateData.status = 'published';
  // 使用传入的 publishedAt 或当前时间
  updateData.publishedAt = publishedAt ? new Date(publishedAt) : new Date();
  ...
```

CLI 端的发布链路一般长这样：

```
CLI get_pending_posts → opencli post 实际发布 → CLI 报告结果
```

**问题场景**：CLI 在"opencli post"阶段死机（断网、OOM、被 kill 等），过了 1 小时才重试联通，重新发布成功后回传的 `publishedAt` 仍是死机前的"计划时间"或"上次执行时间"（CLI 自己的本地时钟）。最终：

- 真实发布时间：今天 11:00（重试成功那一刻）
- NextPost 入库时间：今天 12:00（收到回传）
- `Post.publishedAt`：被写成 11:00（外部回传）

数据观察：今天 2026-06-12 的 `publishedAt` 数据里已经出现"发布时间早于 nextpost 服务端 createdAt 数小时"的异常记录。

### 1.2 目标

让 `Post.publishedAt` **始终反映"NextPost 何时在数据库里把这个帖子标记为已发布"**，即服务端时间，与 CLI 端实际发布时间 / 重试链路完全解耦：

- ✅ 解决 CLI 死机重试导致的时间漂移
- ✅ 解决 CLI 客户端时钟不准 / 时区错乱导致的时间错误
- ✅ 解决多实例 / 多终端同时回传导致的时间不一致
- ✅ 不增加 schema 复杂度
- ✅ 不增加客户端对接复杂度（外部仍然可传 `publishedAt`，但服务端忽略）

### 1.3 非目标（v0.5.3 不做）

- 不新增 `externalPublishedAt` 字段（v0.5.3 不保留外部时间）
- 不改数据库 schema
- 不改 `report_publish_result` 的 MCP schema（保留 `publishedAt` 参数以维持向后兼容，仅忽略其值）
- 不改其他工具
- 不改 Web UI
- 不改 `/api/posts` REST 接口

> **设计权衡**：如果未来真的需要"外部平台实际发布时间"作为业务字段（如跨平台发布效果分析），再加 `externalPublishedAt` 列即可，v0.5.3 故意保持小改动。

## 2. 接口设计

### 2.1 MCP 工具层（接口不变，仅语义变化）

`report_publish_result` 的 `inputSchema` 完全保留，`publishedAt` 字段描述改为**仅作为参考，服务端不会写入**：

```json
{
  "publishedAt": {
    "type": "string",
    "description": "（已废弃，服务端忽略）保留字段仅为向后兼容。NextPost 在标记 status=published 时始终使用服务端时间写入 Post.publishedAt，避免外部 CLI 时钟漂移 / 死机重试导致的时间不一致。"
  }
}
```

### 2.2 服务端行为变更

| 场景 | v0.5.2 行为 | v0.5.3 行为 |
|---|---|---|
| `status: 'success'`，传 `publishedAt: '2026-06-12T10:00:00Z'` | 写入 `Post.publishedAt = 2026-06-12T10:00:00Z` | 写入 `Post.publishedAt = Date.now()`（服务端时间），**忽略外部回传** |
| `status: 'success'`，未传 `publishedAt` | 写入 `Post.publishedAt = Date.now()` | 写入 `Post.publishedAt = Date.now()`（不变） |
| `status: 'partial'`，传/不传 `publishedAt` | 写入外部时间 / 当前时间 | 写入 `Date.now()`（统一服务端时间） |
| `status: 'failed'` | 不动 `publishedAt` | 不动（不变） |

### 2.3 数据库层

- ✅ Schema 零改动
- ✅ `Post.publishedAt` 字段语义保持不变，**仅"语义明确化"**（之前是"外部回传时间 or 服务端时间"，现在统一是"服务端标记时间"）

### 2.4 客户端兼容

- ✅ 旧客户端继续传 `publishedAt`：不会报错，字段被静默忽略（**不是 error**）
- ✅ 新客户端可不再传 `publishedAt`：服务端默认行为已正确
- ✅ 不破坏任何现有 happy path

## 3. 实现清单

### 3.1 `src/mcp/external/tools.ts`

**改动点**（最小化，只改 `reportPublishResult` 函数）：

```ts
case 'success':
  updateData.status = 'published';
  // v0.5.3：始终使用服务端时间，避免外部 CLI 死机重试 / 时钟漂移导致的时间不一致
  updateData.publishedAt = new Date();
  if (externalPostId) updateData.externalPostId = externalPostId;
  if (externalPostUrl) updateData.externalPostUrl = externalPostUrl;
  updateData.publishError = null;
  break;
case 'partial':
  updateData.status = 'published';
  // v0.5.3：同上，统一服务端时间
  updateData.publishedAt = new Date();
  updateData.publishError = errorMessage || 'Partial success';
  break;
```

**TOOLS 数组中 `report_publish_result` 的 inputSchema 描述**也对应调整：`publishedAt` 描述改为"已废弃，服务端忽略"。

### 3.2 不改动的部分

- ✅ Prisma schema 不动
- ✅ 数据库不迁移
- ✅ 其他 MCP 工具（`list_accounts` / `get_pending_posts` / `get_post_detail` / `create_post` / `update_post` / `upload_media_from_*`）零改动
- ✅ Web UI 零改动
- ✅ `/api/posts` REST 接口零改动
- ✅ 认证 / Scope 机制零改动
- ✅ `TOOLS` 数组的 `required` 列表不动（`publishedAt` 本来就不是 required）

## 4. 验收标准

### 4.1 单元测试（vitest）

在 `tests/mcp/tools-coverage.test.ts` 的 `describe('MCP Tools - report_publish_result (executeTool)')` 中**更新现有 4 个用例 + 新增 2 个用例**：

| ID | 用例 | 期望 |
|---|---|---|
| TC-UNIT-1（改） | `success` 传 `publishedAt` 旧值 | `Post.publishedAt` 是 `Date.now()`，**不是** `2026-06-01T10:00:00Z` |
| TC-UNIT-2（不变）| `success` 未传 `publishedAt` | `Post.publishedAt` 是 `Date.now()` |
| TC-UNIT-3（改） | `partial` 传 `publishedAt` 旧值 | `Post.publishedAt` 是 `Date.now()`，**不是** `2026-06-01T10:00:00Z` |
| TC-UNIT-4（不变）| `partial` 未传 `publishedAt` | `Post.publishedAt` 是 `Date.now()` |
| TC-UNIT-5（新） | **时间漂移场景**：`success` 传 `publishedAt = 1小时前`，断言 `Post.publishedAt >= beforeCall` | 服务端时间，不是外部回传 |
| TC-UNIT-6（新） | **空字符串 / 无效 ISO 也不报错**：`publishedAt: ''` / `publishedAt: 'not-a-date'` 不影响 `Date.now()` 写入 | 健壮性 |

**关键断言模板**：
```ts
const beforeCall = new Date()
// ... executeTool('report_publish_result', { ... publishedAt: '2026-06-12T10:00:00Z' ... })
const afterCall = new Date()
const publishedAt = postMock.update.mock.calls[0][0].data.publishedAt as Date
expect(publishedAt.getTime()).toBeGreaterThanOrEqual(beforeCall.getTime())
expect(publishedAt.getTime()).toBeLessThanOrEqual(afterCall.getTime())
// 关键：不是外部传的时间
expect(publishedAt.getTime()).not.toBe(new Date('2026-06-12T10:00:00Z').getTime())
```

## 6. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 业务方真的需要"外部平台实际发布时间" | 低 | 当前没有任何业务场景消费这个字段；未来需要时加 `externalPublishedAt` 列（向后兼容追加）即可 |

| 客户端报错"publishedAt 没生效" | 极低 | 字段描述中明确说"已废弃"即可 |
