# V0.6.0 完整清理 — Prisma 残留移除 + DB 路径中立化

> **状态**: 已完成（基础架构部分）
> **目标版本**: v0.6.0
> **基础版本**: v0.5.3
> **完成日期**: 2026-06-19
> **目标**: 解决 v0.5.3 之前的"Prisma 残留"问题，让第三方用户拿到代码不被 prisma/dev.db / prisma/schema.prisma 误导

---

## 1. 背景与目标

### 1.1 现状（v0.5.3）

v0.5.3 实际上已经把 ORM 从 Prisma 迁到 Drizzle（业务代码用 `getDb()` + Drizzle API），但**仓库里仍残留**：

- `prisma/schema.prisma` — 死代码，业务代码不 import
- `prisma/migrations/20260604060924_add_post_title/` — 历史 Prisma migration
- `prisma/prisma/dev.db.backup-20260604-183707` — 嵌套路径的备份文件
- `prisma/` 这个目录名本身 + 内部 `prisma/dev.db` 数据库文件名 — **强烈暗示"还在用 Prisma"**

加上工具链缺口：

- 没有 `drizzle.config.ts`（第三方用户改 schema 时找不到标准入口）
- `seed.ts` 只支持本地 libsql，**不支持 Cloudflare D1**（首次部署后用户得手动 SQL）
- 文档里写 `pnpm drizzle-kit migrate` / `prisma generate` / `prisma db push` 等过时命令
- 单元测试大量 `vi.mock('@/lib/prisma', ...)`，但 `@/lib/prisma` 模块根本不存在（mock 是 noop，导致业务代码走真 DB 互相污染）

### 1.2 目标

让第三方用户从仓库 clone 下来：

1. 看不到任何 `prisma/` 痕迹
2. 一眼明白这是"Drizzle + libsql/D1"项目
3. `pnpm db:*` 脚本覆盖所有场景（generate / migrate / seed / studio）
4. 本地和 CF 部署的 seed 都有统一入口
5. 文档告诉用户三种部署路径该选哪个

## 3. 第三方用户体验

### 3.1 本地首次启动

```bash
git clone https://github.com/Semonxue/NextPost.git
cd nextPost
pnpm install
pnpm db:generate       # 从 src/lib/db/schema.ts 生成 SQL
pnpm db:migrate:local  # 推 schema 到 data/nextpost.db
pnpm db:seed           # 注入 3 个默认 platform
pnpm dev               # 启动 dev server
```

打开 http://localhost:3456 注册账号即可。

### 3.2 部署到 Cloudflare

```bash
# 1. CF Dashboard 创建 D1 + R2（参考 docs/CLOUDFLARE_DEPLOY.md）
# 2. wrangler.jsonc 填 database_id
# 3. 推送代码 → GitHub Actions 自动 deploy
# 4. 首次部署后必做：
pnpm db:migrate:remote   # wrangler d1 migrations apply nextpost-db --remote
pnpm db:seed:d1          # 注入 platform seed 到远程 D1
```

### 3.3 自托管到自有服务器

参见 [DEPLOYMENT.md § 3](./DEPLOYMENT.md#3-自有服务器自托管)。

## 4. 已知问题 / 后续工作

### 4.1 测试覆盖（部分 mock 失效，待 v0.6.1）

**当前状态**：
- 单元测试：480 passed / 80 failed（85.7% 通过率）
- E2E 测试：5 个文件迁移到 shim，状态未知（需要 `pnpm test:e2e` 跑验证）
- 覆盖率：未跑（推断 ≥ 80%，因为大部分 lib/ 业务代码都过测试）

**失败原因**：

10 个 `tests/api/*.test.ts` 文件**业务代码已用 Drizzle，但测试用 Prisma 风格的具名 mock**（`mockFindMany.mockResolvedValue(...)`）。这些 mock 不再对应业务代码任何调用路径，业务代码走真 DB：

- "should return user accounts" → 业务代码 `db.select().from(account).all()` 返回 0 条（DB 是空）
- "should create account successfully" → 业务代码正常 create，但 mock 期望 `mockCreate.mockResolvedValue(...)` 被调，永远不调

**修复方案（v0.6.1）**：

把 mock 风格改成行为风格（参考 `tests/mcp/auth-coverage.test.ts` 已有的 pattern）：

```ts
// 旧（Prisma mock 风格）
mockFindMany.mockResolvedValue([{...}])

// 新（真实 DB + 行为断言）
beforeEach(async () => {
  await truncateAllTables();
  await setupTestDb();
  await db.insert(account).values([{...}]);
});
// 断言：response.status === 200, data.length === 2
```

`tests/api/_db-shim.ts` 已经提供了 `setupTestDb` / `getTestDb` / `truncateAllTables` / `mockDb` 一整套基础设施。**剩下 10 个测试文件的迁移是逐个改 beforeEach + 断言**，预计 4-6 小时工作量。

### 4.2 E2E 测试覆盖率

E2E 测试迁移到 shim 后还没跑过（`pnpm test:e2e`），需要后续验证。

### 4.3 自托管的多实例支持

`getDb()` 只识别 D1 + 本地 libsql，不支持 Postgres/MySQL。多实例自托管需要外置 DB（待 v0.7.0+ 实现）。

## 6. 升级指南（已有项目 v0.5.3 → v0.6.0）

```bash
# 1. 拉代码
git pull

# 2. 重新安装依赖（移除 @prisma/client、@types/better-sqlite3）
pnpm install

# 3. 迁移数据库文件位置
mv prisma/dev.db data/nextpost.db 2>/dev/null || true

# 4. 重新生成 migration（如果 v0.5.x 期间有 schema 改动）
pnpm db:generate
pnpm db:migrate:local

# 5. 验证
pnpm dev
```

> **Note**: v0.5.x 期间 schema 变化不大，所以 1 次 migration 通常够用。如果有多次 schema 改动，需要按顺序跑历次 migration SQL。
