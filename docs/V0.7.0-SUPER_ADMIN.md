# V0.7.0 — 超级管理能力 (Super Admin)

> **状态**: 需求规划（待评审）
> **目标版本**: v0.7.0
> **基础版本**: v0.6.0
> **完成日期**: TBD
> **作者**: Mavis
> **目标**: 给 NextPost 加一个根用户（root）层级，登录后能跨用户看到所有用户、账号、平台、帖子以及它们的汇总统计，用于运维观测和问题排查

---

## 1. 背景与目标

### 1.1 现状（v0.6.0）

- `User` 表只有 `username / password / email / aiProvider / aiApiKey / aiModel`，**没有任何 role / isAdmin 字段**
- 认证走 NextAuth Credentials，session 通过 JWT 携带 `user.id`
- 所有 API（accounts / posts / stats / settings …）都按 `eq(table.userId, session.user.id)` 隔离，**没有任何跨用户读取入口**
- 侧边栏 `navItems` 是静态数组，普通用户和管理入口目前无法区分

→ 结果：管理员没办法一眼看出"现在有几个用户、谁的账号最多、哪个平台用得最多、最近有没有失败率飙升"，只能直接进 SQLite / D1 拉 SQL。

### 1.2 目标

让运维（部署者 / 多用户场景下的管理员）能用 Web UI 完成这些事：

1. 在 env 里指定 root 凭证，首次启动自动建号
2. root 登录后侧边栏多一个 **「管理」** 菜单
3. 看到用户/账号/平台/帖子的全局列表 + 各自的数据汇总
4. 看到每个账号的「使用情况概要」（帖子数 / 状态分布 / 最近活跃 / 失败率）
5. 不破坏现有数据隔离；普通用户看到的页面 / API 完全不变

### 1.3 v0.7.0 范围（更新）

经过评审，v0.7.0 调整为：

**✅ 做**
- root 能查看所有用户 / 账号 / 平台 / 帖子（只读）
- root 能**管理账号**（跨用户 CRUD：编辑、删除、恢复）——这是 v0.7.0 的写操作焦点
- 所有 admin 写操作记入 `AdminAuditLog` 操作日志
- 提供 `/admin/logs` 页面查看日志

**❌ 不做**（留给 v0.7.x / v0.8.0）
- 创建/删除/封禁**用户**（root 能看用户列表，但不动用户）
- 把普通用户提升为 root
- 「以该用户身份登录」（impersonation）
- 删除/编辑别人的**帖子**
- 系统级设置（全局限流、强制清理缩略图等）

**取舍依据**：账号（Account）是「内容归属单位」，运维中经常需要帮用户修错填的 handle、清理不活跃的账号；帖子是用户的创作成果，删除影响大；用户管理涉及登录凭证和审计，最敏感。三类敏感度递增，v0.7.0 只碰最不敏感的「账号」。

## 2. 关键决策（评审点）

下面 6 个点是这次设计里值得讨论的，我先给**我的建议**，标 ✅，你不同意直接说。

### ✅ 决策 1：root 身份走「env 引导 + DB 持久化」混合模式

不是纯 env（不存 DB，没法参与常规查询），也不是纯 DB（部署时需要手动 seed）。

```
首次启动（无 root 用户）:
  1. 读 env: ROOT_USERNAME, ROOT_PASSWORD
  2. 若 username 在 User 表里不存在 → 创建一个，role='root'
  3. 若已存在但 role != 'root' → 自动升级为 'root'（一次性迁移）
  4. ROOT_PASSWORD 改了 → 下次启动用新密码覆盖（让运维能轮换）
```

后续 root 走普通 NextAuth Credentials 登录，session 里带 `user.role`，所有 admin 路由按 `role==='root'` 判定。

**理由**：env 是部署期可控的"单一信任源"，DB 是运行期"可查询"。两者各取所长。

### ✅ 决策 2：User 表加一个 `role` 列（不用 boolean）

```ts
role: text("role").notNull().default("user")
// 取值: "user" | "root"  （未来可扩展 "admin"）
```

- 用 string 而不是 boolean：将来要加 `admin` / `moderator` 等中间档不用改 schema
- 默认 `"user"`，所有现存用户自动是 user，行为不变
- 唯一索引：**不需要** unique，username 已经是唯一

### ✅ 决策 3：env 配置方式

```bash
# .env（本地开发）+ .dev.vars + GitHub Secrets（部署）
ROOT_USERNAME=admin
ROOT_PASSWORD=<strong-password>
# 可选:
ROOT_EMAIL=admin@example.com
```

- `ROOT_USERNAME` 没设 → **不创建 root**，所有人都看不到管理菜单（部署者可选择不开启此能力）
- `ROOT_PASSWORD` 没设但 `ROOT_USERNAME` 设了 → 启动时 `console.error` 报警，**不启动**
- 这俩走 GitHub Secrets + Cloudflare Dashboard Variables，符合现有 `.env` / `.dev.vars` / `wrangler.jsonc` 三处配置规范

### ✅ 决策 4：管理菜单的可见性判断走 session，不是 env

- 侧边栏读 `session.user.role === 'root'`，为 true 才渲染「管理」菜单
- 中间件不拦截 `/admin/*`（避免多查一次 DB），由每个 admin 页面 server component 自己 `auth()` 判 `role`，失败 redirect `/`
- 普通用户即使手敲 `/admin` 也直接 redirect 走，体验上"从来没存在过这个菜单"

### ✅ 决策 5：admin API 全部走 `/api/admin/*` 前缀

- 命名空间清晰，和现有 `/api/accounts` / `/api/posts` 隔离
- 所有 admin 路由调同一个 `requireRoot()` helper（类似 `auth()`，但失败 throw 403）
- 不复用现有 user-scoped API 后缀加 admin 权限（容易漏判点；显式路由前缀 + helper 双重保险）

### ✅ 决策 6：v0.7.0 admin 在「账号」维度有写操作，用户/帖子维度只读

| 维度 | v0.7.0 权限 |
|---|---|
| 账号 (Account) | **可写**：编辑（name/handle/description/platformId）、软删除、从回收站恢复。审计必记 |
| 用户 (User) | 只读：列表 + 计数；不能改密码、不能封号、不能删除 |
| 平台 (Platform) | 只读：列表 + 配置；不能改 |
| 帖子 (Post) | 只读：跨用户列表 + 筛选 + 分页；不能改、不能删 |
| 操作日志 | 只读（root 看自己和其他 root 的操作） |

写操作涉及审计、责任追溯、误操作回滚——所以即使是「账号」这个相对低敏感的维度，也要：
1. 所有 admin 写操作路由**必须**写 `AdminAuditLog`，失败回滚
2. UI 上加二次确认（"确认删除账号 @xxx？这是用户 Alice 的账号，操作不可逆，但可在 30 天内从回收站恢复"）
3. 不暴露「强制覆盖用户内容」之类的能力——编辑账号只能改 metadata（name/handle/description/platformId），不能改 userId、不能改 createdAt

```ts
// 新增表 AdminAuditLog
export const adminAuditLog = sqliteTable(
  "AdminAuditLog",
  {
    id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
    actorUserId: text("actorUserId").notNull(),       // 操作者（root 的 id）
    action: text("action").notNull(),                  // "account.update" / "account.delete" / "account.restore"
    entityType: text("entityType").notNull(),          // "account" / "user" / "platform" / "post" / "system"
    entityId: text("entityId").notNull(),              // 被操作对象 id
    entityOwnerId: text("entityOwnerId"),              // 被操作对象的拥有者（account.userId 等），便于按"受害者"查
    beforeState: text("beforeState"),                  // JSON.stringify() 之前
    afterState: text("afterState"),                    // JSON.stringify() 之后
    description: text("description"),                  // 人类可读："修改了账号 @xxx 的 handle 从 a 改为 b"
    ipAddress: text("ipAddress"),                      // request.headers.get("x-forwarded-for") 或 "cf-connecting-ip"
    userAgent: text("userAgent"),                      // request.headers.get("user-agent")
    createdAt: text("createdAt").notNull().$defaultFn(() => new Date().toISOString()),
  },
  (t) => [
    index("AdminAuditLog_actor_idx").on(t.actorUserId),
    index("AdminAuditLog_entity_idx").on(t.entityType, t.entityId),
    index("AdminAuditLog_entityOwner_idx").on(t.entityOwnerId),
    index("AdminAuditLog_createdAt_idx").on(t.createdAt),
  ]
);
```

**记录策略**：
- **写操作必记**（account.update / account.delete / account.restore）— 失败 → 整个请求回滚
- **读操作不记**（GET /api/admin/*）— 避免日志被列表请求刷爆
- v0.7.0 唯一写操作是 account.*；用户/平台/帖子的写操作留给后续版本

迁移文件：`drizzle/0001_add_user_role_and_audit_log.sql`（`pnpm db:generate` 生成）。

D1 上：`pnpm db:migrate:remote` 应用。本地：`pnpm db:migrate:local`。

**索引说明**：admin audit log 按时间倒序是热路径（按 createdAt desc）；按 actor 查（"admin 张三最近做了什么"）和按 owner 查（"用户李四的账号被谁动过"）是次热路径，都加索引。

### 3.4 启动期 root 引导（新增）

新文件 `src/lib/admin/bootstrap.ts`：

```ts
// 幂等：每次启动跑一次，根据 env 自动对齐 root
export async function ensureRootUser(db) {
  const username = process.env.ROOT_USERNAME;
  const password = process.env.ROOT_PASSWORD;
  if (!username) return; // 没配就跳过

  if (!password) {
    throw new Error("ROOT_USERNAME 已设置但 ROOT_PASSWORD 未设置，配置错误");
  }

  const existing = await db.select().from(user).where(eq(user.username, username)).get();
  const hashed = await bcrypt.hash(password, 10);

  if (!existing) {
    await db.insert(user).values({
      username,
      password: hashed,
      email: process.env.ROOT_EMAIL ?? null,
      role: "root",
    }).execute();
    console.log(`[admin] Created root user: ${username}`);
  } else if (existing.role !== "root") {
    await db.update(user)
      .set({ role: "root", password: hashed, updatedAt: new Date().toISOString() })
      .where(eq(user.id, existing.id))
      .execute();
    console.log(`[admin] Promoted existing user to root: ${username}`);
  } else {
    // 已是 root，env 改了密码则覆盖（让运维能轮换）
    await db.update(user)
      .set({ password: hashed, updatedAt: new Date().toISOString() })
      .where(eq(user.id, existing.id))
      .execute();
    console.log(`[admin] Refreshed root password from env: ${username}`);
  }
}
```

**调用时机**：

- 不在每个请求都跑（浪费）。在以下时机跑一次：
  1. `scripts/dev.mjs` 启动 dev server 时（已存在入口）
  2. CF Workers 没有"启动钩子"，但可以在 layout 或 layoutRoot 第一次访问时 lazy call。**建议**：在第一个 admin 路由被访问时跑 + 一个 5 分钟内存缓存。
  3. `pnpm db:seed` 也调用一次（保证本地/远程 DB 一致）

**简化**：考虑到 dev 体验和 CF 部署都希望"启动就生效"，v0.7.0 直接在 `src/lib/db/index.ts` 的 `getDb()` 里检查并 run（首次启动会慢 100ms）。

> 这里有取舍：要么 `getDb()` 慢一点，要么首次访问 `/admin/*` 慢一点。我倾向前者（更可预期）。**评审点**。

### 3.7 「使用情况概要」的具体口径

每个账号展示：

```ts
type AccountUsageSummary = {
  accountId: string;
  accountName: string;
  handle: string;
  platformName: string;
  ownerUsername: string;          // 哪个用户的账号
  totalPosts: number;
  byStatus: { draft: number; scheduled: number; published: number; failed: number };
  lastPublishedAt: string | null;  // 最近一次 published 的时间
  lastActiveAt: string | null;     // 最近一次任何 post 的创建/计划/发布时间
  failureRate: number;             // failed / (published + failed)，保留 1 位小数 %
};
```

SQL 一次性 aggregate：

```sql
SELECT
  a.id, a.name, a.handle, p.name as platform, u.username as owner,
  COUNT(post.id) as total,
  SUM(CASE WHEN post.status='draft' THEN 1 ELSE 0 END) as draft,
  SUM(CASE WHEN post.status='scheduled' THEN 1 ELSE 0 END) as scheduled,
  SUM(CASE WHEN post.status='published' THEN 1 ELSE 0 END) as published,
  SUM(CASE WHEN post.status='failed' THEN 1 ELSE 0 END) as failed,
  MAX(CASE WHEN post.status='published' THEN post.publishedAt END) as lastPublishedAt,
  MAX(post.updatedAt) as lastActiveAt
FROM Account a
JOIN Platform p ON a.platformId = p.id
JOIN User u ON a.userId = u.id
LEFT JOIN Post post ON post.accountId = a.id AND post.deletedAt IS NULL
WHERE a.deletedAt IS NULL
GROUP BY a.id
ORDER BY totalPosts DESC;
```

Drizzle 用 SQL 模板或 `db.select(...).from().leftJoin().groupBy()` 表达。

| env 密码暴露 | 部署环境泄漏 → 任何能拿到 env 的人都是 root | 文档明确：ROOT_PASSWORD 必须放 GitHub Secrets / CF Dashboard Variables，不要进 repo；README 加警示框 |

| 启动期 root 引导阻塞 | `getDb()` 慢 100ms | 接受；未来可改成 lazy + cache |
| Sidebar 之前是 module-level 常量 | 改成依赖 session 需要变成 client component（已经是了）+ 重渲染 | 现有 Sidebar 已经是 `"use client"` 且用 `useSession`，直接扩即可 |
| 跨用户查询性能 | 用户/账号/帖子多了之后单表聚合慢 | v0.7.0 用户量预估 < 1000，单 SQL 完全够；以后上 1w+ 再考虑物化视图 |
| ROOT_PASSWORD 被覆盖后旧 session 仍可用 | root 轮换密码后老 JWT 还能登几分钟（JWT 过期前） | 接受；JWT TTL 已是 24h 级别，运维轮换密码时可同时改 `AUTH_SECRET` 强制失效 |
