# V0.5.2 — `get_pending_posts` 时间窗口过滤 — 发布计划

> **状态**: 计划中
> **目标版本**: v0.5.2
> **基础版本**: v0.5.0
> **创建日期**: 2026-06-11
> **目标交付**: `get_pending_posts` 工具新增 `windowMinutes` 时间窗口参数，默认 60 分钟（1 小时）

---

## 1. 背景与目标

### 1.1 现状
v0.5.0 之前的 `get_pending_posts` 没有时间窗口限制，一次返回所有 `status='scheduled'` 的帖子（最多 `limit` 条，按 `scheduledTime ASC`）。在 cron 轮询发布场景下，这会带来两个问题：
1. 客户端必须自己过滤"当前时间附近"的帖子，否则拉到一周后才会发布的帖子没意义
2. 计划帖子数量大时，每次返回很多无用数据，增加网络与解析开销

### 1.2 目标
为 `get_pending_posts` 增加一个**可选参数** `windowMinutes`：
- 以当前服务器时间为中心的**对称窗口** `[now - N min, now + N min]`
- 不传时**默认 60 分钟**（最小化变更原则）
- 仅作用于本工具，其他工具 / API / Web UI / 数据库零改动

### 1.3 非目标（v0.5.2 不做）
- 异步窗口（仅 `[now, now+N]` 不包含过去）
- 全局 MCP 工具时间窗口
- `create_post` / `update_post` 等写工具的时间窗口
- 数据库 schema 改动
- 前端 Web UI 改动

## 2. 接口设计

### 2.1 新增参数

`get_pending_posts` 的 `inputSchema` 增加：

| 字段 | 类型 | 必填 | 默认值 | 范围 | 说明 |
|---|---|---|---|---|---|
| `windowMinutes` | `integer` | 否 | `60` | `0 ~ 43200`（30 天） | 以当前时间为中心的对称时间窗口（分钟） |

### 2.2 行为

**服务端时间**为参考基准，按 UTC 计算。区间：
- `lower = now - windowMinutes * 60_000`
- `upper = now + windowMinutes * 60_000`
- 命中条件：`lower <= scheduledTime <= upper`

Prisma 查询片段（`src/mcp/external/tools.ts`）：
```ts
where.scheduledTime = {
  gte: new Date(lower),
  lte: new Date(upper),
};
```

### 2.3 默认行为变化 ⚠️

| 场景 | v0.5.0 行为 | v0.5.2 行为（默认） |
|---|---|---|
| 不传 `windowMinutes` | 返回所有 `scheduled` 帖子（受 `limit` 限制） | 仅返回 `[now - 60min, now + 60min]` 内的 scheduled 帖子 |
| 显式传 `windowMinutes: 43200` | — | 返回 ±30 天内的 scheduled 帖子（≈ 当前行为） |
| 显式传 `windowMinutes: 0` | — | 仅返回 `scheduledTime === now` 的帖子（一般用不到） |

### 2.4 验证规则

| 取值 | 处理 |
|---|---|
| 不传 / `undefined` | 应用默认 60 |
| `null` | 返回 `INVALID_ARGUMENT` |
| 字符串 / 布尔 / 对象 | 返回 `INVALID_ARGUMENT` |
| `NaN` / `Infinity` | 返回 `INVALID_ARGUMENT` |
| 小数（如 `3.5`） | 返回 `INVALID_ARGUMENT`（必须 integer） |
| 负数（如 `-1`） | 返回 `INVALID_ARGUMENT` |
| `> 43200` | 返回 `INVALID_ARGUMENT` |
| `0 ~ 43200` 整数 | 接受 |

错误响应格式（与其他工具错误一致）：
```json
{ "error": "windowMinutes must be an integer between 0 and 43200", "errorCode": "INVALID_ARGUMENT" }
```

## 3. 实现清单

### 3.1 `src/mcp/external/tools.ts`

**改动点**：

1. 在 `TOOLS` 数组 `get_pending_posts` 的 `inputSchema.properties` 新增：
```ts
windowMinutes: {
  type: 'integer',
  description: '以当前时间为中心的对称时间窗口（分钟），只返回 scheduledTime 落在 [now-N, now+N] 区间内的帖子。取值 0~43200（30 天），默认 60（1 小时）。适合定时轮询发布场景，避免一次返回太多。',
  default: 60,
  minimum: 0,
  maximum: 43200,
}
```

2. 工具 `description` 追加："…不传或省略 `windowMinutes` 时默认 60（±1 小时）。"

3. 新增模块级常量：
```ts
const DEFAULT_WINDOW_MINUTES = 60;
const MAX_WINDOW_MINUTES = 43200;
```

4. 新增 `validateWindowMinutes(value: unknown)` 辅助函数（导出用于测试）。

5. `getPendingPosts` 签名增加 `windowMinutes?: number` 参数（默认 = `DEFAULT_WINDOW_MINUTES`），构造 where 时附加 `scheduledTime: { gte, lte }`。

6. `executeTool` 中 `get_pending_posts` case 解析参数后调用 `validateWindowMinutes`，非法时返回 `INVALID_ARGUMENT`。

### 3.2 不改动的部分

- ✅ Prisma schema 不动
- ✅ 数据库不迁移
- ✅ 其他 MCP 工具（`list_accounts` / `get_post_detail` / `report_publish_result` / `create_post` / `update_post` / `upload_media_from_*`）零改动
- ✅ Web UI 零改动
- ✅ `/api/posts` REST 接口零改动
- ✅ 认证 / Scope 机制零改动

## 6. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 旧客户端期望"返回所有" | 中 | 文档明确默认 60min；客户端可显式传 `43200` 还原旧行为 |
| 默认窗口过短，错过计划时间 | 低 | 用户可显式传更大的 `windowMinutes`（上限 43200） |
| 多个 AI 客户端并发拉取 → 时间窗口边缘帖子重复/漏发 | 低 | 文档建议客户端自己加去重（基于 `post.id`） |
